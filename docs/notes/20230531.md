# In-memory non-relational database on 2D arrays

This is a small project idea: build an in-memory, non-relational database in
Haskell where the storage layer is deliberately simple, but the transaction
layer is interesting.

The goal is not to build a production database. The goal is to understand how
basic storage, transactions, and isolation levels interact when the whole system
is small enough to hold in your head.

## Define Scope

The database stores tables as 2D arrays. Each table is a grid of rows and
columns, and each cell stores a value.

The first version should support:

- creating a table with a fixed schema
- inserting rows
- reading rows by row id
- updating cells
- deleting rows
- starting, committing, and aborting transactions
- running the same operations under different isolation levels

The first version should not support:

- SQL parsing
- indexes
- joins
- persistence
- query planning
- distributed execution
- secondary replicas

Keeping these out of scope is important. The interesting part here is the
transaction behavior, not building every feature expected from a database.

## Setup skeleton and user interfaces

The project can start with a small library and a thin REPL or CLI around it.
The library should expose the database operations directly, and the CLI can
exist only to make manual testing easier.

The rough API can look like this:

```haskell
createTable :: TableName -> [Column] -> Txn ()
insertRow    :: TableName -> [Value] -> Txn RowId
readRow      :: TableName -> RowId -> Txn (Maybe Row)
updateCell   :: TableName -> RowId -> ColumnName -> Value -> Txn ()
deleteRow    :: TableName -> RowId -> Txn ()
```

Transaction execution should make the isolation level explicit:

```haskell
runTxn :: IsolationLevel -> Database -> Txn a -> IO (Either TxnError a, Database)
```

The CLI can stay minimal:

```text
begin serializable
insert users ["utkarsh", 25]
read users 1
update users 1 age 26
commit
```

This gives enough surface area to test reads, writes, conflicts, and rollback
behavior without implementing a full SQL layer.

## Define data models

The storage model can be intentionally boring.

```haskell
newtype TableName = TableName Text
newtype ColumnName = ColumnName Text
newtype RowId = RowId Int
newtype TxnId = TxnId Int

data Value
  = VInt Int
  | VText Text
  | VBool Bool
  | VNull

data Column = Column
  { columnName :: ColumnName
  , columnType :: ColumnType
  }

data Row = Row
  { rowId  :: RowId
  , values :: Vector Value
  }

data Table = Table
  { schema :: Vector Column
  , rows   :: Vector (Maybe Row)
  }

data Database = Database
  { tables :: Map TableName Table
  }
```

Using `Vector (Maybe Row)` keeps row ids stable after deletion. A deleted row can
be represented as `Nothing`, and a later compaction pass can be avoided in the
first version.

The transaction layer needs extra state:

```haskell
data IsolationLevel
  = ReadUncommitted
  | ReadCommitted
  | RepeatableRead
  | Serializable

data TxnState = TxnState
  { txnId       :: TxnId
  , isolation   :: IsolationLevel
  , readSet     :: Set CellRef
  , writeSet    :: Map CellRef Value
  , snapshot    :: Maybe Database
  }
```

`CellRef` can identify a single cell:

```haskell
data CellRef = CellRef TableName RowId ColumnName
```

For a toy implementation, row-level references may be easier than cell-level
references. Cell-level tracking is more precise, but row-level tracking is
simpler and still useful for learning the isolation behavior.

## Implement each isolation level in phases

The four classic isolation levels are:

1. Read uncommitted
2. Read committed
3. Repeatable read
4. Serializable

Each level restricts more behavior than the one before it.

### Phase 1: read uncommitted

This is the weakest isolation level. A transaction can read data written by
another transaction before that transaction commits.

Implementation approach:

- apply writes directly to shared in-memory state
- keep an undo log for rollback
- allow reads to see the latest value, even if it is uncommitted

This phase is useful because it exposes the problem clearly. Dirty reads are
possible by design:

1. transaction A updates a value
2. transaction B reads that value
3. transaction A aborts
4. transaction B has observed a value that never committed

### Phase 2: read committed

Read committed prevents dirty reads. A transaction should only read committed
data.

Implementation approach:

- keep transaction writes private until commit
- reads consult the committed database state
- commit applies the transaction write set atomically

This can be implemented with a global commit lock at first. That is not the most
concurrent design, but it keeps the correctness model simple.

Read committed still allows non-repeatable reads:

1. transaction A reads a row
2. transaction B updates and commits the same row
3. transaction A reads the row again and sees a different value

### Phase 3: repeatable read

Repeatable read ensures that if a transaction reads the same row twice, it sees
the same value both times.

Implementation approach:

- take a snapshot of the committed database when the transaction starts
- serve reads from that snapshot
- keep writes in the transaction write set
- on commit, check for write-write conflicts

This gives a simple MVCC-like model. Reads are stable because they come from the
snapshot. Writes are validated before they become visible.

This level can still allow some anomalies depending on the exact implementation,
especially predicate-related anomalies. Since this toy database does not support
indexes or predicates initially, row-level repeatability is enough for the first
version.

### Phase 4: serializable

Serializable is the strongest level. The outcome should be equivalent to
running committed transactions one at a time in some order.

Implementation approach:

- track the read set and write set for each transaction
- validate at commit time that no committed transaction changed data read by the
  current transaction
- abort and retry when validation fails

This is optimistic concurrency control. The transaction runs without taking
heavy locks, and commit decides whether the transaction was safe.

A transaction should fail validation if:

- another transaction committed a write to something in its read set
- another transaction committed a conflicting write to something in its write
  set

The simplest implementation can use a global version counter. Every committed
cell or row stores the version at which it was last modified. A transaction
records the database version it started from. During commit, it checks whether
any row it read was modified after that start version.

## Testing the behavior

The tests should focus on anomalies, not just CRUD behavior.

Examples worth testing:

- dirty read is possible under read uncommitted
- dirty read is not possible under read committed
- repeated reads can change under read committed
- repeated reads stay stable under repeatable read
- conflicting serializable transactions cannot both commit
- aborting a transaction does not leak its private writes

Small deterministic tests are better than a large random test suite at the
start. Once the semantics are clear, property tests can be added for invariants
like row id stability, rollback correctness, and serializable validation.

## Why Haskell

Haskell is a good fit for this experiment because the boundary between pure
state transitions and effectful execution can be kept explicit. Most of the
database logic can be modeled as transformations over immutable values, while
the outer shell handles concurrency, IO, and the REPL.

That separation should make it easier to test the core transaction logic without
having to spin up an external service or rely on timing-heavy tests.
