# Creating a Personal Website

I wanted a personal website that was simple to write for, easy to maintain, and
boring to deploy. This note captures the setup I use for this site: Markdown
content, MkDocs for generation, Material for MkDocs for the theme, and GitHub
Pages for hosting.

## Static site generator

A static site generator is a good fit for a personal website. There is no
server to run, no database to maintain, and no admin UI to keep updated. The
site is just source files in Git, and the generated output is static HTML, CSS,
and JavaScript.

There are multiple good options for this:

- Jekyll
- MkDocs
- Hugo
- Docusaurus

This repository uses [MkDocs](https://www.mkdocs.org/) with
[Material for MkDocs](https://squidfunk.github.io/mkdocs-material/). MkDocs is
Python based, configured with YAML, and pleasant enough for a notes-first
website. Material adds the parts I do not want to build myself: responsive
layout, search, navigation, dark mode, tags, and decent defaults.

## Local setup

The Python dependencies are pinned in `requirements.txt`:

```text
mkdocs==1.6.1
mkdocs-material==9.7.6
mkdocs-git-revision-date-localized-plugin==1.5.1
mkdocs-glightbox==0.5.2
cairosvg==2.9.0
pillow==12.2.0
```

The `Makefile` keeps the common commands short.

```bash
make setup
```

To run the site locally:

```bash
make run
```

This starts `mkdocs serve`, which serves the website locally and reloads when
content changes.

To build the site once:

```bash
make build
```

This runs `mkdocs build` and writes the generated site to the `site/`
directory.

## Content

The writing flow is intentionally plain. Pages are Markdown files under
`docs/`. The home page is `docs/index.md`, the about page is `docs/about.md`,
and notes live under `docs/notes/` as dated Markdown files.

For example, this note is:

```text
docs/notes/20230527.md
```

A note starts with front matter for tags:

```markdown
---
tags:
  - Website
---
# Creating a Personal Website
```

Tags are configured in `mkdocs.yml`, and the tag index is generated through
`docs/notes/index.md`.

Images are kept under `docs/assets/`. The home page profile image, for example,
is stored at:

```text
docs/assets/images/profile.jpg
```

## Configuration

The site-level configuration lives in `mkdocs.yml`.

The basic metadata is:

```yaml
site_name: Utkarsh Saxena
site_url: https://utk-spartan.github.io/
site_description: Personal website of Utkarsh Saxena
site_author: Utkarsh Saxena
docs_dir: docs/
```

The site uses the Material theme with navigation tabs, section navigation,
search suggestions, search highlighting, dark and light palettes, and social
links.

Useful plugins enabled in this repo:

- `search`, for local site search
- `git-revision-date-localized`, for page creation and update dates
- `tags`, for browsing notes by topic
- `glightbox`, for image previews
- `social`, for social preview cards in CI

## GitHub Pages

GitHub Pages is enough for this kind of site. The repository has a GitHub
Actions workflow at `.github/workflows/main.yml` that deploys the site when
changes are pushed to `master`.

The trigger is:

```yaml
on:
  push:
    branches:
      - master
```

And the deployment steps are:

1. checks out the repository
2. sets up Python
3. installs dependencies with `make setup`
4. deploys the site with `mkdocs gh-deploy --force`

`mkdocs gh-deploy` builds the static site and publishes the generated output to
the GitHub Pages branch.

The live site URL is:

```text
https://utk-spartan.github.io/
```

## Google analytics

Analytics is optional. I have kept the Material for MkDocs analytics section in
`mkdocs.yml`, but commented out:

```yaml
# analytics:
#   provider: google
#   property:
```

To enable Google Analytics, create a Google Analytics property and add the
property ID there.

## Google search indexing

Once the site is deployed, it can be added to Google Search Console using the
site URL:

```text
https://utk-spartan.github.io/
```

After verification, submit the generated sitemap:

```text
https://utk-spartan.github.io/sitemap.xml
```

MkDocs generates the sitemap during the build, so no custom sitemap file is
needed for the basic setup.
