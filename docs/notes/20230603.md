# Causal chains and event ordering in distributed systems

Distributed systems make ordering difficult. On a single machine, it is tempting
to think of events as happening in one clear sequence. Across machines, that
sequence does not really exist.

Clocks drift, messages take different paths, network delays vary, and two
events can happen independently without either one being "first" in a meaningful
way. For many systems, the useful question is not "which event happened earlier
by wall-clock time?", but "could this event have influenced that event?"

That is the idea behind causal ordering.

## Logical clock algorithms

A logical clock is a way to assign timestamps to events without depending on
physical time.

Instead of asking machines to agree on the current time, each process maintains
some metadata that moves forward as events happen. That metadata is attached to
messages and merged when messages are received.

Logical clocks are useful because they describe ordering from the system's own
execution, not from an external clock source.

The most common models are:

- Lamport timestamps
- vector clocks

They answer slightly different questions.

## Lamport timestamps

Lamport timestamps give every event a monotonically increasing number.

Each process keeps a counter:

1. increment the counter before every local event
2. attach the counter value to every outgoing message
3. when receiving a message, set the local counter to `max(local, received) + 1`

This gives a simple ordering rule:

```text
event A happened-before event B => timestamp(A) < timestamp(B)
```

The important detail is that the reverse is not guaranteed. If
`timestamp(A) < timestamp(B)`, it does not necessarily mean A caused B. Lamport
timestamps can tell us that one event may be ordered before another, but they
cannot reliably distinguish causality from unrelated timing.

Example:

```text
Process P1: A -------- send message -------- C
                         |
                         v
Process P2: B -------- receive message ----- D
```

Here, `A` causally influences `D` if the message sent after `A` is received
before `D`. But `B` may be completely independent of `A`, even if its Lamport
timestamp is larger.

Lamport clocks are still useful. They are small, cheap, and enough when a total
or partial processing order is more important than detecting concurrency.

## Vector clocks

Vector clocks preserve more information.

Instead of a single counter, each process keeps a map from process id to
counter:

```text
P1: { P1: 3, P2: 1, P3: 0 }
```

The rules are:

1. increment your own entry before every local event
2. attach the whole vector clock to outgoing messages
3. on receive, take the element-wise maximum of the local and received clocks
4. increment your own entry after merging

Vector clocks let us compare two events:

- `A < B` if every component of `A` is less than or equal to `B`, and at least
  one component is strictly less
- `A == B` if all components are equal
- `A || B` if neither clock is less than the other

The last case means the events are concurrent. Neither event causally depends on
the other.

Example:

```text
A = { P1: 2, P2: 0 }
B = { P1: 2, P2: 3 }
```

`A < B`, because `B` has seen everything `A` had seen, plus more from `P2`.

But:

```text
A = { P1: 2, P2: 0 }
B = { P1: 1, P2: 3 }
```

These are concurrent. `A` knows more about `P1`, while `B` knows more about
`P2`.

## Causal chains of events

A causal chain is a sequence of events where each event could have influenced
the next one.

The basic happened-before relation is:

- events on the same process are ordered by local execution
- sending a message happens before receiving that message
- the relation is transitive

So if:

```text
A happened-before B
B happened-before C
```

then:

```text
A happened-before C
```

This matters in real systems. For example:

- replicated databases need to know whether two writes conflict or whether one
  supersedes the other
- chat systems may want replies to appear after the messages they refer to
- distributed logs need to separate causal order from arrival order
- caches and materialized views need to avoid applying stale updates over newer
  ones

Wall-clock timestamps are often not enough for these cases. A message can arrive
late with an older physical timestamp but still carry causally important state.

## Implement vector clock in scala

The core data structure can be small.

```scala
final case class NodeId(value: String) extends AnyVal

final case class VectorClock private (versions: Map[NodeId, Long]) {
  def increment(node: NodeId): VectorClock = {
    val next = versions.getOrElse(node, 0L) + 1L
    copy(versions = versions.updated(node, next))
  }

  def merge(other: VectorClock): VectorClock = {
    val allNodes = versions.keySet ++ other.versions.keySet

    val merged = allNodes.map { node =>
      val left = versions.getOrElse(node, 0L)
      val right = other.versions.getOrElse(node, 0L)
      node -> math.max(left, right)
    }.toMap

    VectorClock(merged)
  }

  def compare(other: VectorClock): ClockOrdering = {
    val allNodes = versions.keySet ++ other.versions.keySet

    val comparisons = allNodes.map { node =>
      val left = versions.getOrElse(node, 0L)
      val right = other.versions.getOrElse(node, 0L)
      java.lang.Long.compare(left, right)
    }

    val hasLess = comparisons.exists(_ < 0)
    val hasGreater = comparisons.exists(_ > 0)

    (hasLess, hasGreater) match {
      case (false, false) => ClockOrdering.Equal
      case (true, false)  => ClockOrdering.Before
      case (false, true)  => ClockOrdering.After
      case (true, true)   => ClockOrdering.Concurrent
    }
  }
}

object VectorClock {
  val empty: VectorClock = VectorClock(Map.empty)
}

sealed trait ClockOrdering

object ClockOrdering {
  case object Before extends ClockOrdering
  case object After extends ClockOrdering
  case object Equal extends ClockOrdering
  case object Concurrent extends ClockOrdering
}
```

A send event increments the sender clock and attaches it to the message:

```scala
val p1 = NodeId("p1")

val clockAfterSend =
  VectorClock.empty
    .increment(p1)
```

A receive event merges the incoming clock and then increments the receiver:

```scala
val p2 = NodeId("p2")

val receiverClock =
  VectorClock.empty
    .merge(clockAfterSend)
    .increment(p2)
```

Now the receiver event is causally after the sender event:

```scala
clockAfterSend.compare(receiverClock)
// ClockOrdering.Before
```

Concurrent events look different:

```scala
val a = VectorClock.empty.increment(NodeId("p1"))
val b = VectorClock.empty.increment(NodeId("p2"))

a.compare(b)
// ClockOrdering.Concurrent
```

## Trade-offs

Vector clocks are more expressive than Lamport timestamps, but they are also
larger. The timestamp grows with the number of participating nodes.

That is acceptable for small clusters, actor systems, CRDT experiments, and
debugging tools. For very large systems, vector clocks need pruning, version
vectors, dotted version vectors, or a different causality-tracking model.

The useful lesson is still the same: physical time is not the same as causal
time. Once multiple machines are involved, event ordering becomes part of the
data model.
